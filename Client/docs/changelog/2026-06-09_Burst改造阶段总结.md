# Burst 改造阶段总结

## 1. 本次会话目标

本轮工作的目标不是直接把项目改成 ECS，而是按风险最低的顺序推进第一阶段优化：

- 先做真机 Profiler 采样定位热点。
- 先处理 Battle.HandleUnits 中最明显的 GC 和数据结构问题。
- 在不改玩法逻辑的前提下，把热路径数据逐步改造成更适合 Burst 的形式。
- 先落地 Burst，再考虑更进一步的 JobSystem 和 ECS。

这次改造的核心原则是：先把热路径数据扁平化，再让真正高频的路径数学计算进入 Burst。

## 2. 过程回顾

### 2.1 项目级分析与方向确认

- 对当前项目做了 DOTS 引入可行性扫描。
- 结论是 Battle.cs 的战斗模拟层最适合作为第一阶段切入点。
- 用户确认优先级为 Burst -> Jobs -> ECS，而不是一开始就做整套 ECS 重构。

### 2.2 包依赖准备

- 在 Packages/manifest.json 中引入了 Burst 和 Collections。
- 当前仓库里还存在 Entities 相关包，但本阶段没有实际开始 ECS 迁移。

### 2.3 真机采样与热点确认

- 在 Battle.cs 和 UI_Battle.cs 中加入了 ProfilerMarker。
- 真机采样后，Battle.HandleUnits 被确认是当前稳定交火场景下的主要热点。
- 热点不仅体现在 CPU 时间，也体现在 GC.Alloc 和 GC.Collect。

### 2.4 热路径 GC 减压

围绕 Battle.HandleUnits 做了第一轮不改变玩法行为的减压：

- 去掉部分 LINQ、临时集合拼接和不必要的中间结构。
- 把高频搜索和排序逻辑改为手写顺序遍历。
  具体来说，原来目标选择这块更偏向“先把候选目标拼起来，再做筛选或排序”，现在改成了几组专门的顺序扫描函数：
  - TryGetClosestTarget 负责在单个候选字典里直接扫描最小距离目标。
  - UpdateClosestTarget 和 TryGetClosestAllTarget 负责跨 otherTargets、resourceTargets、defenceTargets 三组候选做顺序比较，不再为了找最近目标额外构造合并结果。
  - FillSortedTargets 把三组候选目标复用写入 Battle 内部的 sortedTargetBuffer，再原地排序，避免每次进打墙逻辑都重新创建一组新集合。
- 降低目标选择和路径决策流程中的托管分配。
  这一步的重点不是把所有逻辑都变成 O(1)，而是把热路径里的“临时容器创建”改成“复用已有容器 + 单次遍历”：
  - AssignTarget 直接接收已有候选字典并立刻做最近目标决策，不再为中间结果生成新的字典或列表。
  - GetPathToWall 直接消费 sortedTargetBuffer 里的候选顺序，边遍历边判断有没有必要先转成打墙路径。
  - GetPathToBuilding 在比较常规路径和无视阻挡路径时，直接用点数组做长度和挡墙统计，而不是围绕对象链表再包一层临时结构。

### 2.5 路径数据扁平化

为后续 Burst 做准备，路径系统先从引用对象链表逐步改造成值类型结果：

- AStarSearch 新增了直接返回 Vector2Int 路径点的方法。
  具体做法是保留 Search 内部的 Cell.Parent 回溯逻辑，但在搜索结束后增加了一层 BuildLocationPath，把整条路径一次性拷贝成 Vector2Int[]，并通过 FindLocations 暴露给 Battle 使用。这样 Battle 拿到的是连续坐标数据，而不是 A* 的内部节点对象。
- Vector2Int 补充了 Equals / GetHashCode / IEquatable 支持。
  这一步是为了让路径点从“仅仅能装坐标的 struct”变成“可安全比较、可做键值判断、可放进后续 Native 容器”的值类型。路径有效性校验、起终点比较，以及后续 NativeArray 承载都依赖这类稳定的值语义。
- Battle.Path 不再依赖 A* 内部 Cell 链来长期持有路径。
  以前 Battle 侧更依赖 A* 搜索阶段生成的对象关系；现在改成：
  - Path.Create 直接接收 Vector2Int 点序列。
  - Path 自己保存 start、end、pointCount 和 points。
  - GetPathLength、CountBlockingWalls、PopulateBlockingWalls 这些后续逻辑都围绕点序列工作，而不是沿着 Cell.Parent 链二次回溯。
  这样做的直接好处是，Battle 的运行时路径数据已经从“引用很多 A* 内部对象”变成了“只保存一段扁平坐标序列”，后面再切到 NativeArray 和 Burst 时，改动面会小很多。

### 2.6 实际 Burst 落地

本次会话的最终落地点，是让 Battle 运行时真正使用 Burst 处理路径数学。

最开始尝试过把 Burst 数学内核写成直接调用的静态方法，签名为 NativeArray<Vector2Int> -> float / position；
但 Unity Burst 的直接调用路径不接受这种签名，因此出现了 Unity 控制台里的 BC1064 报错。

最终采用的可编译落地方案是：

- 将路径持久存储改为 NativeArray<Vector2Int>。
- 使用 BurstCompile 的 IJob 作为 Burst 执行载体。
- 通过 Battle 级别的持久 scratch buffer 复用输出 NativeArray，避免每次计算都临时分配结果容器。
- 让 GetPathLength 和 GetPathPosition 通过 Burst Job 包装执行。

## 3. 当前已完成的代码改造

### 3.1 Battle.cs

- 增加 ProfilerMarker 采样点。
- 增加单位路径释放逻辑 ClearPath()。
- 在目标切换、治疗目标切换、单位死亡、路径结束等位置补充路径释放。
- 增加 Battle.DisposeRuntimeData() 统一释放运行时 NativeArray 数据。
- Battle.Path.points 已从托管数组切换为 NativeArray<Vector2Int>。
- 路径长度和路径插值已改为通过 Burst Job 执行。
- 增加 Battle 级复用 scratch buffer：
  - 一个 NativeArray<float> 用于路径长度输出
  - 一个 NativeArray<BurstPathPosition> 用于路径位置输出

### 3.2 UI_Battle.cs

- 在关闭战斗时释放 Battle 的运行时 NativeArray。
- 在重新创建 Battle 前先释放旧实例的运行时数据。

### 3.3 AStarSearch.cs

- 支持直接返回值类型路径点集合。
- 减少 Battle 层对 A* 内部对象链的长期依赖。

### 3.4 Vector2Int.cs

- 增加值类型比较与哈希支持。
- 为路径结果的值类型使用和后续 Native 容器承载打基础。

## 4. 当前技术状态

截至本次会话结束，第一阶段已经从“Burst 准备”进入“Burst 实际运行时接入”状态。

当前已成立的链路如下：

1. A* 仍然负责寻路。
2. Battle.Path 长期持有的路径点已经是 NativeArray<Vector2Int>。
3. 单位移动时的路径长度计算与路径位置插值，已经通过 BurstCompile 的 Job 执行。
4. UI_Battle 生命周期已接入 NativeArray 清理，避免战斗关闭和重建时泄漏。

## 5. 本次修复的关键问题

本轮最后修复的就是 Unity 控制台中的 Burst 报错。

### 问题现象

- Missing BurstCompile attribute on type Battle containing bursted method
- Burst error BC1064: Unsupported parameter NativeArray<Vector2Int>
- Burst error BC1064: Unsupported return type BurstPathPosition

### 根因

- 直接 Burst 调用的静态方法不支持当前这组 NativeArray<Vector2Int> 参数和自定义返回值签名。
- 之前把 Burst 内核写成 Battle 内部静态方法时，触发了 Burst 直接调用的签名限制。

### 最终修法

- 去掉直接 Burst 调用的静态方法模式。
- 改为 BurstCompile 的 IJob 封装执行。
- 用 Battle 内部持久 scratch buffer 复用输出容器。
- 保留 NativeArray 路径数据，不回退到托管路径表示。

## 7. 这一阶段的收益与边界

### 已获得的收益

- Battle 热路径中的托管分配明显比初始状态更少。
- 路径数据已经进入 Native 容器形态。
- 部分高频路径数学已经进入 Burst 执行通道。
- 生命周期清理已补齐，避免 NativeArray 泄漏。

### 仍然存在的边界

- A* 内部节点结构仍是传统对象模型，不是 Native 化节点池。
- 当前只是 Burst 第一阶段，并没有做并行 Job 化寻路或完整 ECS 化。
- 真正更大的收益，仍然需要后续把 A* 内部状态继续改成 struct + Native 容器。

## 8. 建议的下一步

最值得优先做的事情不是继续盲改，而是回到真机做同场景复测。

建议继续在 60 单位稳定交火场景下，重点复看：

- Battle.HandleUnits 的 CPU 时间
- 单帧 GC.Alloc
- GC.Collect 频率
- 路径移动相关采样点是否变得更稳定

如果第二轮真机数据仍然显示 HandleUnits 里路径和寻路部分占比很高，那么下一阶段最合理的方向就是：

- 把 A* 的 Cell / open set / closed set 改造成 struct + Native 容器
- 之后再考虑是否将寻路评估迁移到更明确的 Job 化执行
- 最后才考虑把战斗实体管理逐步迁到 ECS

## 9. 一句话结论

这次会话已经完成了 Burst 第一阶段里最关键的一步：

不是停留在“为 Burst 做准备”，而是已经让 Battle 的一段真实运行时热路径开始使用 Burst，并且补齐了 NativeArray 的生命周期管理。
