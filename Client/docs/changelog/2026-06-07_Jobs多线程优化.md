## 三、Phase A — Grid Epoch O(1) 重置

### 改动原因
**发现：** `AStarSearch.Reset()` 每次调用都会遍历 49×49=2401 个 `Cell` 对象清零 G/H/F/Closed/Parent，
而 `GetPathToBuilding` 最多调用 4角×2套Grid = 8 次 A*，即每个单位重寻路时最多 8×2401≈19208 次字段清零。
Grid.Reset 的 O(2401) 是全表遍历，但 A* 实际只访问几十到几百个 cell——大量清零是无效的。

### 核心原理
用 **Epoch（纪元）** 替代全量清零：
- 每轮搜索递增一个整数 epoch
- Cell 被访问时检查 `cell.SearchId == epoch?` → 不等则原地清零并打上当前 epoch 戳
- 未访问的 cell 永远不被清零

### 具体改动

**文件 1：** `Assets/Scripts/AStarPathfinding/Cell.cs`
- 新增字段 `public int SearchId;`（L18）

**文件 2：** `Assets/Scripts/AStarPathfinding/AStarSearch.cs`
- 新增字段 `private int _searchEpoch = 0;`（L32）
- **新增方法 `EnsureCellFresh(Cell cell)`**（~L95–L107）：首次访问时 O(1) 延迟清零
- **`Reset()` 重写**（~L115–L119）：从 `_grid.Reset()` + `_open.Clear()` 改为 `_searchEpoch++` + `_open.Clear()`（O(1)）
- **`Search()` 方法**：在起点和每个邻居节点首次接触前调用 `EnsureCellFresh()`；移除旧的 `_grid[neighbour.Location].Closed` 直接引用（改为 `neighbour.Closed`）
- **删除 `BuildCellPath`、`CountPathNodes(Cell)`、`Find`**：仅保留 `FindLocations`（返回 `Vector2Int[]`）

**文件 3：** `Assets/Scripts/AStarPathfinding/Grid.cs`
- `Reset()` 添加注释说明此方法仅构造函数中调用一次，不再被搜索流程调用

### 效果量化
| 指标 | 旧方案 | Epoch 方案 |
|------|--------|------------|
| Reset 复杂度 | O(2401) 固定 | O(1) |
| 每 A* 调用 cell 清零数 | 2401 | 实际访问量（~50–300）|
| GetPathToBuilding(8 A*) 清零数 | 19208 | ~1600 |

## 四、Phase B — Burst A* IJob（核心重构）

### 改动原因
Phase A 解决了 O(n²) 重置问题，但 A* 算法本身仍在 **托管堆** 上运行：
- `Cell` 是 class（引用类型），GC 可追踪
- `FastPriorityQueue` 操作 class 数组
- G/H/F 使用 `double`（Burst 对 float 更友好）
- `Vector2Int[]` per-call 堆分配
- 受限于 C# 托管执行速度

将其改写为 `[BurstCompile] IJob`：
- 全部数据放在 `NativeArray<T>`（堆外内存，零 GC）
- 运算由 Burst 编译器 SIMD 优化
- 后续可无缝升级为多线程 Schedule

### 具体改动

**文件 1（新建）：** `Assets/Scripts/AStarPathfinding/BurstAStarJob.cs`（~260 行）

| 组件 | 实现 |
|------|------|
| `BurstNodeData` struct | `float G/H/F`、`int ParentIndex/HeapIndex/SearchId`、`byte Closed` — 全 blittable |
| `AStarBurstJob : IJob` | 完整 A* 实现：epoch 延迟初始化、最小堆增删改、8 方向展开、切角阻断、路径回溯 |
| 索引方式 | flat `y * gridW + x`（替代 `Cell[,]`） |
| 堆实现 | `NativeArray<int>` 1-indexed 数组堆，`heap[0]=count` |
| 启发函数 | Octile Distance（`float` 版） |
| 输出 | `NativeArray<Vector2Int> resultPath` + `NativeArray<int> resultLength` |

**文件 2：** `Assets/Scripts/Battle.cs`

**B2 — 持久 Scratch Buffer（2 套）**

Burst Job 只能读写 `NativeArray<T>`（堆外非托管内存），**不能访问托管堆上的 `Cell[,]` 对象**。
因此必须把寻路依赖的"哪些格子被建筑阻挡"这份数据，从 `grid[x,y].Blocked`（C# bool 字段，托管内存）
复制到 `NativeArray<byte>` 中，供 `AStarBurstJob.Execute()` 读取。

```csharp
// 网格大小：49×49 = 2401 格，flat 索引 y * gridW + x
//   burstSearchBlocked[i] = 1  表示格子 i 被建筑占据，地面单位不可通过
//   burstSearchBlocked[i] = 0  表示格子 i 空闲，可以行走
NativeArray<byte> burstSearchBlocked;     // [gridW*gridH]，0=可走 / 1=阻挡

// burstUnlimitedBlocked 始终全零 —— 飞行单位和"忽略墙体评估"模式完全无视阻挡
NativeArray<byte> burstUnlimitedBlocked;  // [gridW*gridH]，始终全零
```

**为什么需要两套？**

游戏中有两种寻路模式，对应两个 `AStarBurstJob` 实例：

| 模式 | 阻挡数据 | 使用者 | 用途 |
|------|---------|--------|------|
| **search**（遵守阻挡） | `burstSearchBlocked` | 地面单位（步兵、骑士等） | 按真实墙体布局寻路 |
| **unlimited**（忽略阻挡） | `burstUnlimitedBlocked`（全零） | 飞行单位、破墙评估 | 无视墙体直飞 / 计算"如果没墙，路径多长"以判断是否值得打墙 |

同一个 tick 内这两种 Job 会并行 Schedule，各自读取独立的 blocked 数组，互不干扰。

**数据流向：**

```
游戏初始化（Initialize）
  ├─ grid[x,y].Blocked = true          ← 建筑放置时标记格子
  └─ burstSearchBlocked[y*W+x] = 1     ← 同步写入 NativeArray 副本
      （burstUnlimitedBlocked 始终全零，无需写入）

建筑被摧毁（DamageBuilding → SyncBurstBlockedForBuilding）
  ├─ grid[x,y].Blocked = false         ← TakeDamage 内部清除
  └─ burstSearchBlocked[y*W+x] = 0     ← 同步清除 NativeArray 副本

每帧寻路（ScheduleBurstFindLocations → FlushBurstRequests）
  └─ AStarBurstJob.Execute()
       └─ if (blocked[nextIndex] != 0) continue;  ← 直接读 NativeArray，零跨内存域开销
```

**为什么不用原来的 `grid`？**

`grid` 是 `Grid` 类对象，内部存的是 `Cell[,]`（class 二维数组）。
Burst 编译器**拒绝**任何托管引用类型 —— `Cell` 是 class，包含 `Cell Parent` 引用链和 `bool Closed` 等字段，全都不符合 Burst 的 blittable 要求。
因此必须将唯一需要跨边界传递的数据（阻挡状态）抽离为一个扁平的 `NativeArray<byte>`。

生命周期：`EnsureBurstAStarBuffers()` 分配（`Allocator.Persistent`） → `DisposeRuntimeData()` 释放

**B3 — blocked 状态同步**
- `Initialize()` 中 `grid[x,y].Blocked = true` → 同步写 `burstSearchBlocked[y*W+x] = 1`
- `DamageBuilding()` 包装方法：`TakeDamage` 后若 health≤0 → `SyncBurstBlockedForBuilding()` 清除 blocked
- 替换全部 3 处直接 `TakeDamage` 调用为 `DamageBuilding()` 包装

**B4 — BurstFindLocations 辅助**
```csharp
private int BurstFindLocations(bool unlimited, Vector2Int start, Vector2Int goal)
```
组装 `AStarBurstJob` → `.Run()` → 返回路径点数

**B5 — Path 类 NativeArray 重载**
- `Path.IsValid(NativeArray<Vector2Int>, int, Vector2Int, Vector2Int)` 重载
- `Path.Create(NativeArray<Vector2Int>, int, BattleVector2Int, BattleVector2Int)` 重载（内部 `NativeArray.Copy`）

**B6 — 替换全部调用点**
- `GetPathToBuilding` 地面段：`search.FindLocations`/`unlimitedSearch.FindLocations` → `BurstFindLocations(false/true, ...)`
- `GetPathToBuilding` 破墙段：同上
- `GetPathToBuilding` 飞行段：同上
- `GetPathToWall`：同上
- `path.Create(ref search, ...)` → `BurstFindLocations` + `path.Create(NativeArray, ...)`

**B7 — 移除兼容层**
- 删除字段 `unlimitedGrid`、`search`、`unlimitedSearch`
- 删除 `Path.Create(ref AStarSearch, ...)` 重载
- 删除 `search`/`unlimitedSearch` 初始化代码

**文件 3：** `Assets/Scripts/AStarPathfinding/AStarSearch.cs`
- 注释全量替换为 `[已废弃]` 标记，指向 `BurstAStarJob.cs`

## 五、Phase C — Scratch Pool + 批量并行 Schedule

### 改动原因
Phase B 的 `.Run()` 在调用线程**同步串行**执行——每个 `BurstFindLocations` 调用都会阻塞主线程直到该次 A* 完成，
然后才处理下一个。在 GetPathToBuilding 地面主循环里（最多 8 次 A*），这 8 次是**一个一个串行跑完**的。

Phase C 改造成：
1. 先收集所有请求（只 queue，不执行）
2. 一次性 `.Schedule()` 全部到 Worker Thread 上**并行**运行
3. `.Complete()` 阻塞等待全部完成（**不是异步回调，不会跨帧**）
4. 逐条读取结果

### 关键澄清：没有用 async/await，`Complete()` 是同步阻塞

```
Phase B (Run)                          Phase C (Schedule + Complete)
────────────                           ─────────────────────────────
Run()  → 主线程同步执行 A*              Schedule() → Worker Thread 执行
Run()  → 主线程同步执行 A*              Schedule() → Worker Thread 执行
Run()  → 主线程同步执行 A*              Schedule() → Worker Thread 执行
Run()  → 主线程同步执行 A*              ...
                                       Complete() → 主线程阻塞，等全部完成
总耗时 ≈ 8 × 单次 A* 时间              总耗时 ≈ max(单次 A* 时间) + 调度开销
                                       ↑ 依然是同步的，ExecuteFrame() 返回前
                                         所有结果都已就绪，确定性不变
```

`JobHandle.Complete()` 类似于 `Task.WaitAll()`——**阻塞当前线程直到全部 Job 执行完毕**。
没有回调、没有跨帧等待、没有不确定性。战斗循环 20 ticks/sec 的同帧确定性完全保留。

### 关于 Pool 容量：16 槽 vs 300 单位同时寻路

**Pool 不是并发限制。** 实际并行度由 Unity Job Scheduler 根据 CPU 核数（如 8 核）自动管理——
`Schedule()` 300 个 Job 和 `Schedule()` 8 个 Job 一样，多余的 Job 会在引擎内部排队，不会丢。

Pool 的真正作用是**内存预分配（免 GC）**，不是流量控制。它为什么 16 槽就够？

```
HandleUnit(单位1) → GetPathToBuilding → Schedule 8 个 → Flush → 池回收（8 槽归还）
HandleUnit(单位2) → GetPathToBuilding → Schedule 8 个 → Flush → 池回收
HandleUnit(单位3) → ...
...
HandleUnit(单位300) → 同上
```

每次 `FlushBurstRequests` 后 `_astarPoolNextSlot = 0`，池**立即回收全部槽位**。
同一时刻池里最多只有**一个 GetPathToBuilding 的 8 个请求**（4 角 × 2 搜索）。
300 个单位串行调用 GetPathToBuilding，每次用 8 个槽，用完归还，周而复始——16 槽绰绰有余。

真正限制并行粒度的不是 Pool，而是当前架构下 GetPathToBuilding 是**per-unit 串行调用**的。
如果要实现跨单位的全帧级并行（300 个单位的所有 A* 请求一次性 Schedule），需要把 Flush 推迟到
ExecuteFrame 末尾，但那样 GetPathToBuilding 的破墙分支逻辑（依赖 A* 结果做判断）会变得非常复杂。

### 核心设计（三方法流水线）

```
                    Phase A: Queue                    Phase B: Execute             Phase C: Read
                    ═══════════                      ════════════                 ════════════
                    
ScheduleBurstFindLocations()               FlushBurstRequests()            GetBurstResult()
  ↓                                           ↓                               ↑
借一个 pool slot                            遍历 _pendingAStar              从 pool slot
记 start/goal/unlimited                     为每个构建 AStarBurstJob        读回 resultPath +
入 _pendingAStar list                       全部 .Schedule()               resultLength
返回 slot int (不阻塞)                       JobHandle.Combine()            供后续 Path.Create()
                                              .Complete() ← 同步阻塞
                                              清空 _pendingAStar
```

### 具体改动

**C1 — Pool 字段声明**

```csharp
const int ASTAR_POOL_SIZE = 16;                       // 内存预分配槽数，非并发上限

// 4 个 NativeArray 数组，每个 slot 一套独立缓冲区（互不干扰，可并行读写）
NativeArray<BurstNodeData>[] _astarPoolNodes;         // A* 搜索临时状态 [nodeCount] per slot
NativeArray<int>[]           _astarPoolHeaps;         // 最小堆 [heapLen] per slot
NativeArray<Vector2Int>[]    _astarPoolResultPaths;   // 结果路径 [nodeCount] per slot
NativeArray<int>[]           _astarPoolResultLengths; // [0]=路径点数，-1=无路 per slot

int[]       _astarPoolEpochs;     // 每个 slot 各自的 epoch 计数器
int         _astarPoolNextSlot;   // 轮转借出指针

// 请求队列（per-batch，每次 Flush 后清空）
struct PendingAStar { int slot; bool unlimited; int2 start, goal; }
List<PendingAStar> _pendingAStar;
```

> 外层是 managed 数组（`NativeArray<T>[]`），其元素才是 `NativeArray<T>`。
> 外层数组的分配只在 `Initialize()` 发生一次（64 个槽位），不会在每帧产生 GC。

**C2 — Pool 生命周期**

| 方法 | 调用时机 | 作用 |
|------|---------|------|
| `EnsureAStarPool(nodeCount, heapLength)` | `Initialize()` → `EnsureBurstAStarBuffers()` | 一次性预分配 64 套 `Allocator.Persistent` NativeArray |
| `DisposeAStarPool()` | `DisposeRuntimeData()` | 战斗结束时释放所有 slot |
| 每帧 `FlushBurstRequests()` | 多次，每批调度后 | `_pendingAStar.Clear()` + slot 归零 |

**C3 — ScheduleBurstFindLocations（替换旧 BurstFindLocations）**

```csharp
// Phase B 旧版 — 立即 .Run()，阻塞主线程
private int BurstFindLocations(bool unlimited, Vector2Int start, Vector2Int goal)
{
    new AStarBurstJob { ... }.Run();          // ← 同步，此处阻塞
    return burstSearchResultLength[0];        // ← 结果直接从全局变量读
}

// Phase C 新版 — 只 queue，不执行
private int ScheduleBurstFindLocations(bool unlimited, Vector2Int start, Vector2Int goal)
{
    int slot = _astarPoolNextSlot;            // 借一个独立 slot（互不干扰）
    _astarPoolNextSlot = (slot + 1) % 64;
    _pendingAStar.Add(new PendingAStar {      // 记入请求队列，不执行任何计算
        slot = slot, unlimited = unlimited,
        start = new int2(start.X, start.Y),
        goal = new int2(goal.X, goal.Y),
    });
    return slot;                              // 返回 handle，调用方暂存
}
```

**C4 — FlushBurstRequests + GetBurstResult**

```csharp
private void FlushBurstRequests()
{
    if (_pendingAStar.Count == 0) return;

    // 为每个 pending 请求创建独立 Job（各自 pool slot，互不干扰）
    var handles = new NativeArray<JobHandle>(_pendingAStar.Count, Allocator.Temp);
    for (int i = 0; i < _pendingAStar.Count; i++)
    {
        var req = _pendingAStar[i];
        _astarPoolEpochs[req.slot]++;
        handles[i] = new AStarBurstJob
        {
            blocked      = req.unlimited ? burstUnlimitedBlocked : burstSearchBlocked,
            nodes        = _astarPoolNodes[req.slot],        // 每个 slot 独立
            heap         = _astarPoolHeaps[req.slot],
            resultPath   = _astarPoolResultPaths[req.slot],
            resultLength = _astarPoolResultLengths[req.slot],
            // ...
        }.Schedule();  // ← 不阻塞，立即返回 JobHandle
    }

    JobHandle.CombineDependencies(handles).Complete();  // ← 同步阻塞，等全部完成
    handles.Dispose();
    _pendingAStar.Clear();
    _astarPoolNextSlot = 0;
}

private void GetBurstResult(int slot, out int pointCount, out NativeArray<Vector2Int> path)
{
    pointCount = _astarPoolResultLengths[slot][0];      // -1 = 无路
    path       = _astarPoolResultPaths[slot];            // 有效时 pointCount 个 Vector2Int
}
```

**C5 — 各调用场景总结**

| 调用场景 | Phase B 耗时 | Phase C 耗时 | 策略 |
|---------|-------------|-------------|------|
| `GetPathToBuilding` 地面主循环（4角×2搜索） | 8×串行 | ≈max(8)×并行 + 调度 | collect → 1×Flush → 遍历 |
| `GetPathToBuilding` 飞行段（4角） | 4×串行 | ≈max(4)×并行 + 调度 | collect → 1×Flush → 遍历 |
| `GetPathToBuilding` 破墙段（2 次） | 2×串行 | ≈max(2)×并行 | schedule → flush → get |
| `GetPathToWall` | 按组串行 | 按组并行 | schedule → flush → get |
| `ExecuteFrame` 末尾 | — | `FlushBurstRequests()` 兜底 | 确保无遗漏 |

移除 10 个被 Phase B 使用但 Phase C 不再需要的旧 scratch 字段：
`burstSearchNodes`, `burstUnlimitedNodes`, `burstSearchHeap`, `burstUnlimitedHeap`,

## 六、架构现状（改造完成后）

```
Battle.ExecuteFrame()
  └─ HandleUnits 循环
       └─ HandleUnit (per unit, 串行)
            └─ target < 0 → FindTargets → AssignTarget

grid                              ← 仅存 blocked 状态
burstSearchBlocked / burstUnlimitedBlocked ← Burst A* 读取的阻挡数组
AStarSearch.cs / FastPriorityQueue.cs     ← 已废弃，保留作算法参考
```

## 九、FAQ：为什么 Unity Jobs / Burst 必须用 NativeArray 而不能用托管类型？

### 问题
> 引入 Job 和多线程是必须要 Native 类型的容器吗？托管类型（`List<T>`、`Dictionary<K,V>`、`class` 对象）不行？是因为线程不安全吗？

### 答案：三个独立原因，线程安全只是其中之一

#### 原因 1：Burst 编译器只接受 blittable 类型

Burst 是一个 **AOT（Ahead-of-Time）编译器**，将 C# Jobs 代码直接编译为高度优化的机器码。
它**完全不能处理托管对象（class）**，因为：
- `class` 实例分配在 GC 堆上，由垃圾回收器管理，地址可能随时变动
- `List<T>` 内部是 class 数组，同样在 GC 堆上
- `Dictionary<K,V>` 内部是复杂的哈希表结构，包含多个 class 引用
- Burst 能处理的只有 **blittable（纯值类型、无引用）** 的数据，以及 Unity 专门设计的 `NativeArray<T>` / `NativeList<T>` 等容器

```csharp
// ❌ Burst 直接拒绝：
List<int> managedList;          // 托管对象
Dictionary<int, float> dict;    // 托管对象
Cell cell;                      // class 引用类型

// ✅ Burst 可以：
NativeArray<int> nativeArray;   // 非托管容器
float3 position;                // 纯值类型
```

#### 原因 2：Unity Jobs 安全系统禁止访问托管内存

即使不用 Burst（纯 `IJob`），Unity 的 Job System 也有**硬性规则**：
- Job 中**不能**访问任何托管对象（class、delegate、string 等）
- 只能在 Job 字段中声明 `NativeArray<T>` 或其他 `NativeContainer` 子类
- 编译器会在 Job 编译时执行静态分析，拒绝违规代码

这是 Unity 的**安全设计**，不是性能优化——目的是防止你在多线程中不小心共享了被 GC 移动的对象导致崩溃。

#### 原因 3：线程安全由 NativeContainer 的 Safety System 保证

NativeContainer 不仅仅是"能放进 Job"——它们还内置了**线程安全追踪**：

| 机制 | 作用 |
|------|------|
| 读写追踪 | `[ReadOnly]` 标记的 NativeArray 允许多个 Job 同时读；未标记的同一时刻只能被一个 Job 写 |
| 别名检测 | 防止同一个 NativeArray 被作为两个不同字段传入同一个 Job，导致自我数据竞争 |
| Dispose 追踪 | 确保 NativeArray 在所有 Job 完成前不会被释放 |

托管容器（`List<T>` 等）**完全没有这些保护**——两个线程同时 `list.Add()` 会直接导致数据损坏，而且没有任何运行时检测。

### 总结

|  | 托管类型（class / List / Dictionary）| NativeArray |
|--|-----------------------------------|-------------|
| Burst 能编译？ | ❌ | ✅ |
| 能放入 Job 字段？ | ❌ | ✅ |
| 线程安全追踪？ | ❌ | ✅ |
| 内存位置 | GC 堆（可移动）| 堆外原生内存（固定地址）|
| 跨线程共享 | 危险，无保护 | Safety System 自动检测冲突 |
| 本项目中举例 | `Cell[,]`、`List<Tile>`、`FastPriorityQueue(Cell[])` | `NativeArray<byte>`、`NativeArray<BurstNodeData>` |
