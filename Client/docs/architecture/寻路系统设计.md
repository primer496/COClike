# Battle.cs 寻路系统深度分析

## 1. 战斗帧 vs 渲染帧

| 层级 | 频率 | 周期 | 职责 |
|------|------|------|------|
| **战斗帧**（Battle.ExecuteFrame） | 20 ticks/s | **50ms** | 确定性模拟：索敌、寻路、伤害结算 |
| **渲染帧**（Unity Update） | 60fps | **~16.67ms** | 读取 `Unit.position` 做视觉表现 |

**两者完全解耦。** 渲染层不等待战斗帧，直接读 `Unit.position`——该值是战斗帧通过 Burst 加速的路径插值（`PathPositionJob`）连续更新的，所以画面仍然丝滑。Supercell 系游戏（COC/CR/BS）战斗 tick 都是 15~30Hz，这是行业验证过的设计。

## 2. 寻路调用链

```
ExecuteFrame()
  └─ HandleUnits:
       ├─ 第一趟: HandleUnit_MoveAttack (移动+攻击，不触发寻路)
       │    └─ target < 0 的单位 → 进入 _batchPathUnitQueue
       │
       ├─ ProcessBatchPathfinding (跨单位批量寻路)
       │    └─ 一次 FlushBurstRequests，并行执行所有 pending A*
       │
       └─ 第二趟: HandleUnit_MoveAttack (拿到路径的单位立即执行移动)
```

```
FindTargets(unit, priority)
  └─ AssignTarget(unit, wallsPriority)
       ├─ [wallsPriority=true]
       │    └─ GetPathToWall(unit)       ← 优先破墙策略
       │         ├─ 找到墙 → return（直接打墙）
       │         └─ -1 → 继续往下
       │
       └─ GetPathToBuilding(min, unit)   ← 通用建筑寻路
            └─ 途中遇墙也会返回墙索引
```

## 3. GetPathToWall — 破墙寻路

**目的：** 当地面单位到目标的路径被城墙堵死时，找出应该先打哪堵墙及到达路径。

### 流程

```
FillSortedTargets → 合并三类目标并按距离排序
  │
  └─ foreach target (由近到远):
       ├─ slot1 = ScheduleBurstFindLocations(false, start, goal)
       ├─ FlushBurstRequests → GetBurstResult(slot1)
       │
       ├─ [路径通] → continue (跳过，找下一堵墙)
       │
       └─ [路径被堵] →
            ├─ 收集共享友军: 遍历 _units，找已在打同一 mainTarget 墙的友军
            ├─ 对每个友军: 从友军位置寻路到本单位
            │    └─ [可达 + 组队半径内] → 复用友军的破墙路径 ✅
            │
            └─ [友军都不可用] → unlimited A* 寻路
                 └─ 沿路径逐格扫描，找第一个 alive 墙 Tile
                 └─ 截断路径到墙前 → 返回 (墙索引, 截断路径) ✅
```

### 关键细节
- **`wallSearchPath` 通畅就 `continue` 丢弃**：通畅说明不需要打墙，直接试下一个目标
- **共享破墙**：如果友军已经在打同一目标前的墙，本单位的路径可以直接复用友军的 `path.blocks`，省去重新分析墙体

## 4. GetPathToBuilding — 通用建筑寻路

**目的：** 为单位算出到目标建筑的最佳路径，同时判断途中是否需要破墙。

### 流程

```
排除墙/装饰物/障碍物
  │
  ├─ 计算建筑包围盒 corners（1×1=1角，多格=4角）
  │
  ├─ 对每个 corner 同时调度两类 A*：
  │    ├─ burstSearchBlocked (遵守墙)
  │    └─ burstUnlimitedBlocked (无视墙 → 用于数墙数)
  │
  ├─ FlushBurstRequests 批量执行
  │
  └─ 遍历所有 corner 结果，评分选出最优路径:
       评分标准: pathLength 更短 → blockingWalls 更少
       │
       ├─ [blocks.Count > 0] → 路径上有墙
       │    ├─ 尝试共享友军破墙路径
       │    └─ [没有友军] → 截断到最后一个墙 Tile
       │    └─ 设置 mainTarget = buildingIndex
       │    └─ 返回 (墙索引, 截断路径)
       │
       └─ [blocks.Count == 0] → 无墙
            └─ 返回 (buildingIndex, 最优路径)
```

### 飞行单位简化路径
飞行单位只用 `burstUnlimitedBlocked` 做单次搜索，取最短路径，不做墙判断。

## 5. 核心容器

| 容器 | 类型 | 层级 | 职责 |
|------|------|------|------|
| `_buildings` | `List<Building>` | Battle | 所有建筑（含城墙） |
| `_units` | `List<Unit>` | Battle | 所有单位 |
| `grid` | `Grid` (53×53) | Battle | 仅维护 Blocked 布尔状态，不参与搜索 |
| `blockedTiles` | `List<Tile>` | Battle | 扁平阻挡格列表，供 `CountBlockingWalls`/`PopulateBlockingWalls` 遍历 |
| `burstSearchBlocked` | `NativeArray<byte>` | Battle | 1D 阻挡栅格，**wall=blocked** → 传给 Burst A* 做"遵守墙"搜索 |
| `burstUnlimitedBlocked` | `NativeArray<byte>` | Battle | 1D 阻挡栅格，**wall≠blocked** → 传给 Burst A* 做"无视墙"搜索 |
| `sortedTargetBuffer` | `List<KV<int,float>>` | 复用缓冲 | 合并三类候选目标并按距离升序 |
| `sharedWallUnitIndexBuffer` | `List<int>` | 复用缓冲 | 已在打同一墙的友军索引 |
| `sharedWallUnitPositionBuffer` | `List<BattleVector2Int>` | 复用缓冲 | 友军网格位置 |
| `sharedWallPathStartBuffer` | `List<Vector2Int>` | 复用缓冲 | 友军路径起点 |
| `sharedWallSlotBuffer` | `List<int>` | 复用缓冲 | 友军 A* 槽位号 |

### 单位关键字段
| 字段 | 类型 | 含义 |
|------|------|------|
| `target` | `int` | 当前实际攻击目标（可能是墙索引） |
| `mainTarget` | `int` | 最终意图目标建筑索引（当 target 是墙时指向后方建筑） |
| `path` | `Path` | 当前路径（含 points + blocks） |
| `resourceTargets` | `Dictionary<int,float>` | 资源类建筑候选 |
| `defenceTargets` | `Dictionary<int,float>` | 防御类建筑候选 |
| `otherTargets` | `Dictionary<int,float>` | 其他建筑候选 |

## 6. Path 类

```csharp
public class Path {
    public BattleVector2Int start;       // 路径起点（网格坐标）
    public BattleVector2Int end;         // 路径终点（网格坐标）
    public NativeArray<Vector2Int> points;  // 路径点序列
    public int pointCount;               // 有效点数
    public float length;                 // 路径总长度（世界单位）
    public List<Tile> blocks;            // 路径上碰到的墙体 Tile 列表
}
```

- `Create(result, start, end)` — 从 A* 结果复制路径点（使用 `Allocator.Persistent`）
- `IsValid(points, start, end)` — 验证首尾点与起终点匹配
- `ReversePoints()` — 翻转路径方向
- `Dispose()` — 释放 NativeArray

## 7. A* 槽位池（Slot Pool）机制

### 为什么用 Slot？

Burst Job **不能即时返回结果**——它是异步的。必须先攒起所有寻路请求，一次性 `Schedule` + `Complete`，再逐个读取。Slot 就是"先登记、后取货"的货架编号。

### 工作流程

```
ScheduleBurstFindLocations(unlimited, start, goal)
  ├─ 领槽: slot = _astarPoolNextSlot
  ├─ 指针前移: _astarPoolNextSlot = (slot+1) % 512
  ├─ 登记: _pendingAStar.Add({slot, unlimited, start, goal})
  └─ 返回 slot（票据）

FlushBurstRequests()
  ├─ 遍历 _pendingAStar
  ├─ 每个请求创建 AStarBurstJob，传入对应的 _astarPoolNodes[slot]/Heaps[slot] 等
  ├─ JobHandle.CombineDependencies → Complete (阻塞等待全部完成)
  ├─ _pendingAStar.Clear()
  └─ _astarPoolNextSlot = 0 (重置指针)

GetBurstResult(slot, out pointCount, out path)
  └─ 从 _astarPoolResultLengths[slot] 和 _astarPoolResultPaths[slot] 读结果
```

### 池子结构

```
ASTAR_POOL_SIZE = 512

_astarPoolNodes[0..511]       // 每槽 ~53² 个 BurstNodeData
_astarPoolHeaps[0..511]       // 每槽 ~(53²+2) 个 int (堆)
_astarPoolResultPaths[0..511] // 每槽 ~53² 个 Vector2Int
_astarPoolResultLengths[0..511] // 每槽 1 个 int
_astarPoolEpochs[0..511]      // 搜索世代号，防止读到旧数据
```

每槽约占用 50KB（Persistent 分配），512 槽总计约 25MB。

## 8. 溢出与限流

### 会溢出吗？

当前每单位平均 ~6 次 A*（飞行单位少，1×1 建筑少），512 槽约支持 80 个单位同时寻路。如果超过：
- slot 号从 511 绕回 0，后到的请求覆写前面槽位
- 读到的路径无效 → `target` 保持 -1
- **下一战斗帧自动重试**

### 隐式流量整形

- 并非所有单位每帧都丢目标（建筑逐步被摧毁，目标丢失是渐进的）
- 溢出不会崩溃，只是部分单位延迟 1 tick（50ms）再拿到路径
- 对 20Hz 模拟频率，这在绝大多数场景下足够

### 优化方向（非扩池）
- 路径复用：相邻单位共享路径
- 缓存有效路径
- 显式分帧：一帧只处理前 N 个，剩余顺延（当前靠溢出隐式实现）

## 9. target vs mainTarget 分离

```
target     = 当前实际攻击对象（可能是墙）
mainTarget = 单位真正意图攻击的建筑

例：单位想打金矿，但被墙挡住
  target     = wallIndex    （先打墙）
  mainTarget = goldMineIndex（打完墙后打金矿）

墙破后 HandleUnit_MoveAttack 检测到：
  _buildings[mainTarget].health > 0 → 重新寻路到金矿
```
